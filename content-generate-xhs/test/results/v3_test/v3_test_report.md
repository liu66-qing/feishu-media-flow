## 三、问题根因分析

### 3.1 test_05（学习就业）块模式排版（2/3轮出现）

**现象**：模型在处理"学习和就业"这类高信息密度话题时，倾向使用「块内单换行+块间双换行」的混合模式（7-10个大块，每块2-4行），而非要求的22-32个纯双换行段落。

**根因分析**：
1. 该话题信息密度高，模型在压缩信息量时倾向紧凑排版
2. r2成功产出26段完美排版，说明模型**有能力**做到，只是不稳定
3. 可能是因为4个干货要点各自需要展开，模型在"一个要点一个块"和"一个要点拆3段"之间犹豫

**结论**：不是Prompt能力不足，而是稳定性问题。可以通过增加"干货类话题段落拆分示例"或在代码层做后处理（将块内`\n`替换为`\n\n`）来解决。

### 3.2 LLM双重转义bug（1/15）

**现象**：r1_t02的body中`\n`被双重转义为字面`\\n`文本，导致整篇无换行。

**根因**：LLM在生成JSON时偶尔会将换行符转义为`\\n`（在JSON值中这是合法的，因为JSON字符串中`\n`表示换行，但模型多加了一层转义变成`\\n`）。

**解决方案**：这是代码鲁棒性问题，应在`main.py`的`_parse_llm_response()`中增加后处理步骤：检测到body中包含字面`\n`文本时，自动替换为真正的换行符。

### 3.3 标题数字不一致（1/15）

**现象**：r3_t01标题写"4条建议"但正文展开5个要点。

**根因**：当段落数增加到24段、每个要点拆成3段时，模型容易在拆点时多出一个细分点（如将"现场执行"拆为"体验区+互动+气场"三个子点），但忘记更新标题中的数字。

**结论**：自检清单中已有此项检查，但执行率不是100%。可在自检中强化"在确定标题数字前先确定正文要点数量"的顺序约束。

## 五、v4 改进建议（如继续迭代）

### 高优先级（代码层）

1. **增加双重转义后处理**：在`_parse_llm_response()`中检测body是否包含字面`\n`文本，自动替换为真正换行符。预计可解决7%的格式异常。
2. **块内换行自动修复**（可选）：检测到段落数<20但有大量单`\n`时，将单`\n`替换为`\n\n`，可解决test_05类问题。

### 中优先级（Prompt层）

3. **标题数字顺序约束**：在自检清单中要求"先数正文要点数，再确定标题数字"，减少数字不一致。
4. **干货类话题few-shot示例**：增加一个学习/就业类话题的few-shot示例，展示高信息密度内容如何拆分为22-32段。
5. **cover_text多样性约束**：在Prompt中鼓励cover_text使用差异化表达，避免同一话题反复生成相似文案（如"社恐也能轻松交朋友吗"出现2次）。

### 低优先级

6. 正文中"最""绝对""必须"等词的使用可以进一步减少（当前语义上大多合理）。
7. 可考虑增加封面文案的情绪词多样性（"避坑指南""真心话"出现频率较高）。
