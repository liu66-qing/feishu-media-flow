### 问题汇总

1. **test_02 严重格式问题**：LLM 将换行符双重转义（输出 `\\n\\n` 字面文本而非真正换行符），导致整篇 body 成为1个长段，渲染时会出现字面 "\n\n" 文字。JSON 合法但正文格式损坏。**根因**：LLM 偶发的转义输出问题，需在 `_parse_llm_response()` 中增加 `\\n` → `\n` 的后处理。
2. **test_05 段落模式偏差**：使用「块内单换行 + 块间双换行」的混合模式，`\n\n` 分隔仅7段（22-32段要求），块内用 `\n` 分隔句子形成7个中等块，每个块2-4行。视觉上仍有换行呼吸感，但段落间距不够突出，块首过渡词不够显眼。
3. **test_02 标题长度**：3个标题长度 15/15/17，均在12-25范围内，实际合格（标记⚠️仅因正文转义问题影响整体）。

## 三、综合结论

- **硬指标通过率**：10/12（83%）—— 段落数2项不达标（test_02转义bug、test_05块模式偏差），其余全部通过
- **格式合规率**：4/5（80%）—— test_02转义bug为偶发LLM问题
- **肉眼均分**：20.8/25
- **风格合规率**：100%（无emoji编号、无营销腔、全部第一人称）

**本轮结论**：v3 Prompt 在段落节奏上显著改善，3/5篇完美达标22-24段（r1标准下v2仅约1/3达标）。但暴露两类问题：
1. **LLM偶发转义bug**（test_02）：需在解析层增加 `\\n` → `\n` 的后处理修复
2. **学习就业话题(test_05)持续偏好块内换行模式**：三轮测试(v2 r3、v3 r1)均出现此现象，模型在干货密度高时倾向用紧凑排版

**下轮优化方向**：
1. 在 `main.py` 的 `_parse_llm_response()` 中增加双重转义修复逻辑
2. 观察 round2-3 中 test_05 是否仍出现块模式问题（如反复出现，考虑针对干货类话题单独强化段落约束）
