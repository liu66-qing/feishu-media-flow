# 刘俊清：主流程集成与流量反馈闭环

本文所称三个平台固定指：公众号、抖音、小红书。

## 负责目标

把素材采集、平台偏好、内容生成、人工审核、平台交付和发布后复盘连接成完整流程，并通过真实发布数据研究哪些内容更容易获得曝光和互动。

这部分负责保证各成员成果不是独立模块，而是能够在同一条业务链路中持续运行。

## 当前已有基础

- 已有飞书消息、卡片动作和内容创建入口。
- 已有AgentLoop统一状态管理。
- 已有公众号、抖音和小红书的内容生成与媒体交付流程。
- 已有公众号草稿交付、抖音图文人工交付和小红书发布调用。
- 已有账号表和发布日志表的配置入口。
- 已有发布成功、失败和人工接管状态。

当前主流程还没有完整保留热点素材信息，也没有发布内容的帖子链接、多个时间点的表现数据和流量反馈结果。

## 具体任务

- 将自动采集的素材接入飞书素材审核和内容生成流程。
- 保证管理员采纳选题后，标题、来源、建议角度、热度和参考素材都能保留下来。
- 将平台偏好结果应用到对应平台的内容任务中。
- 保证公众号、抖音和小红书按照当前明确的交付边界运行。
- 记录每条内容使用的素材、平台偏好版本和发布时间。
- 为每条发布内容建立可以持续追踪的记录。
- 回收曝光、阅读或播放、点赞、评论、收藏、转发和涨粉等表现数据。
- 观察发布后1小时、6小时、24小时和72小时的表现变化。
- 分析标题、封面、内容结构和发布时间与流量表现的关系。
- 组织单变量对照，避免一次改变多个因素后无法判断原因。
- 形成每周流量复盘，并把可靠结论反馈给下一轮平台偏好分析。
- 区分平台公开规则、样本统计规律和待验证假设，不把平台黑盒机制描述成确定事实。
- 维护项目总进度说明，让README与仓库真实实现保持一致。

## 交付内容

- 素材采纳到内容生成的完整链路。
- 平台偏好进入内容和画面生成的完整链路。
- 每条发布内容的来源、平台、时间和追踪信息。
- 发布后多个时间点的表现记录。
- 每周平台流量复盘。
- 已验证规律与待验证假设的区分说明。
- 与仓库实际进度一致的README和人员分工文档。

## 完成标准

- 管理员采纳素材后，完整信息能够进入内容生成。
- 生成任务能够使用对应平台的偏好结果。
- 每条已发布内容都有明确的平台、时间和追踪记录。
- 发布后能够形成1小时、6小时、24小时和72小时的数据快照。
- 每周能够输出有数据依据的流量复盘。
- 已验证的表现规律能够影响下一轮平台偏好和内容生产。
- 任一成员都能从README和分工文档判断当前进度与自己的任务边界。

## 协作关系

- 接收郝乐瑾提供的真实素材和推荐选题，并保证完整进入主流程。
- 接收尹羿璇提供的平台偏好和生成对照结果，并组织真实发布验证。
- 向尹羿璇反馈发布表现，作为更新平台偏好的依据。
- 负责处理跨目录协作和主链冲突，避免成员分别修改同一业务入口。

## 不在本任务范围内

- 不代替郝乐瑾完成平台素材采集和样本清洗。
- 不代替尹羿璇制定平台语言与画面偏好。
- 不以主观经验直接得出平台流量规律，所有结论必须经过数据验证。
